# Role Routing Protocol

Use this protocol whenever a feature, policy, or product decision is proposed.

## 1. Classify The Decision

- Product scope
- UX flow
- Matching/ranking
- Trust/safety
- Data/impact
- Engineering architecture
- Launch operations
- Legal/policy

## 2. Assign Primary And Review Roles

Every decision has one primary role and at least one review role.

Examples:

- New event ranking signal: Sphere Algorithm Designer primary, Trust Architect review.
- Organizer onboarding: Community Operations Lead primary, Legal Advisor and Engineer review.
- Handprint progression: UX Designer primary, Data Agent review.
- Paid event promotion: Product Strategist primary, Legal Advisor and Trust Architect review.

## 3. Produce A Decision Note

Each decision note should include:

- Context
- Recommendation
- Reasoning
- Risks
- Open questions
- Owner
- Date

## 4. Revisit Triggers

Reopen the decision when:

- Pilot data contradicts the assumption.
- Safety incidents emerge.
- Organizer supply falls below launch density.
- Users do not understand the handprint.
- Legal/policy scope changes.
